- Accept optional `cliPath` parameter with default (per injectable path pattern from storage.ts line 33 — enables testing with non-existent paths). Per D-04, this function must never throw to the caller.
- **Phase 1 — existence check:** `try { await fs.access(cliPath); } catch { ... }`. In the catch block:
  - Call `vscode.window.showInformationMessage("This Code: CLI not found at ~/.this-code/bin/this-code", "Download")`
  - If the user clicks "Download", call `vscode.env.openExternal(vscode.Uri.parse(CLI_DOWNLOAD_URL))`
  - `return;` early (do not proceed to version check)
- **Phase 2 — version check:** `try { const { stdout } = await execFileAsync(cliPath, ["--version"], { timeout: 3000 }); } catch { ... }`.
  - Parse stdout with regex: `const match = stdout.trim().match(/(\d+)\.\d+\.\d+/);`
  - If match found, `const majorVersion = parseInt(match[1], 10);`
  - If `majorVersion !== EXPECTED_CLI_MAJOR`, call `vscode.window.showWarningMessage(...)` with the message: `` `This Code: CLI major version mismatch. Extension expects v${EXPECTED_CLI_MAJOR}.x, found v${majorVersion}.x. Some features may not work correctly.` ``
  - In the catch block (execFile failed — corrupt binary or wrong arch): call `vscode.window.showWarningMessage("This Code: CLI found but could not run \`this-code --version\`. Try reinstalling the CLI.")`

**Key constraints:**
- The `{ timeout: 3000 }` option on `execFileAsync` prevents the version check from hanging indefinitely (Pitfall 5 from RESEARCH.md)
- Both catch blocks show notifications but never throw — per D-04 non-blocking contract and storage.ts error-swallow pattern
- The function returns `Promise<void>` — caller uses `.catch(() => {})` for fire-and-forget
