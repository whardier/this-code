Create `.github/workflows/cli-ci.yml` with the following content:

```yaml
name: CLI CI

on:
  push:
    paths:
      - 'cli/**'
      - '.github/workflows/cli-ci.yml'
  pull_request:
    paths:
      - 'cli/**'
      - '.github/workflows/cli-ci.yml'

jobs:
  build:
    name: Rust CLI (${{ matrix.os }})
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os:
          - ubuntu-latest
          - macos-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install Rust stable
        uses: dtolnay/rust-toolchain@stable
        with:
          components: clippy, rustfmt

      - name: Cache Cargo registry and build artifacts
        uses: actions/cache@v4
        with:
          path: |
            ~/.cargo/registry/index/
            ~/.cargo/registry/cache/
            ~/.cargo/git/db/
            cli/target/
          key: ${{ runner.os }}-cargo-${{ hashFiles('cli/Cargo.lock') }}
          restore-keys: |
            ${{ runner.os }}-cargo-

      - name: Check formatting
        working-directory: cli
        run: cargo fmt --check

      - name: Clippy (deny warnings)
        working-directory: cli
        run: cargo clippy --all-targets -- -D warnings

      - name: Build (release)
        working-directory: cli
        run: cargo build --release

      - name: Test
        working-directory: cli
        run: cargo test
```

Design notes:
- `dtolnay/rust-toolchain@stable` is the recommended action for Rust toolchain setup (preferred over `actions-rs/toolchain` which is unmaintained). It pins to stable and installs clippy + rustfmt components.
- `actions/cache@v4` caches `~/.cargo/registry` (downloaded crates) and `cli/target/` (compiled artifacts) keyed by OS + Cargo.lock hash. This avoids re-downloading `rusqlite` (which bundles and compiles SQLite) on every run.
- `fail-fast: false` ensures both ubuntu and macos jobs report independently — a macOS failure does not cancel the ubuntu job.
- `paths:` filter ensures the workflow only runs when CLI files change. Extension-only changes do not trigger CLI CI.
- No Windows job: PLAT-02 is best-effort; Windows CI is deferred to Phase 4. Do NOT add windows-latest to the matrix.
- Steps ordering: fmt → clippy → build → test (fast failures first; format and lint are cheaper than a full build).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → crates.io | Crate downloads during CI build |
| CI cache → runner | Restored cargo registry and build artifacts from cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06-01 | Tampering | Cargo cache poisoning | accept | GitHub Actions cache is scoped to the repository and keyed by Cargo.lock hash; a changed Cargo.lock invalidates the cache; no cross-repo cache sharing |
| T-02-06-02 | Tampering | Supply chain: crate substitution | mitigate | Cargo.lock pins all transitive dependencies to exact versions and checksums; Cargo verifies checksums on download; committed Cargo.lock prevents version float |
| T-02-06-03 | Denial of Service | rusqlite bundled SQLite compile time | accept | Cache stores `cli/target/` so SQLite is only compiled on cache miss; acceptable CI cost |
</threat_model>
